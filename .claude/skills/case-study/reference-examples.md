# Case Study Reference Examples — Jonathan Schummers

Curated case-study examples Jonathan likes the format of. Used as **format corpus** by the future `/case-study` skill when generating Section 3 (refined narrative) of mission MD files.

Source: external (other designers' portfolios Jonathan admires). To be used as **structural / visual reference**, NOT as content reference (the projects are not Jonathan's).

---

## Format pattern observed across all 3 examples

All 3 examples share:

- **Context block** with 4 visual-anchored sub-blocks using emoji icons:
  - `⁉️ Problem:` (1-2 sentences max)
  - `🎯 Target audience:` (1 line)
  - `👯 Team:` (one-liner listing each role and count)
  - `🔑 Key results:` (bullet list of hard outcomes)
- **`How I solved this problem?`** — numbered list, each item is a single short action statement (usually 1 line, occasionally with a parenthetical clarification)
- **`What did we produce in the end?`** — bullet list of concrete deliverables (sometimes preceded by a single intro sentence)

### Key differences vs Jonathan's previous v2 template

| Aspect | Old v2 template | New compact format |
|---|---|---|
| Context structure | H3 sub-headings (`### Problem`, `### Target audience`, `### My role`) | Inline bold + emoji anchors |
| Key results placement | At the end only (`What we delivered`) | **Up front in Context** (triple-placement reinforced) |
| Steps in "How I solved" | 50-120 words/step | **1 line/step**, occasionally parenthetical |
| Headline + Subtitle | First sections | **Absent** (or rendered at page level outside the body) |
| Visual rhythm | Dense paragraphs | Scannable, emoji-anchored, 60-second readable |

### Generation order implication (validated 2026-05-29)

Counter-intuitive but right: **don't generate Headline + Subtitle first**. You can only write a tight, accurate Headline once the Context + How + Delivered are locked. The new workflow:

1. **Context** (Problem + Target audience + Team + Key results)
2. **How I solved this problem?** (5-9 step outline → per-step content)
3. **What did we produce in the end?**
4. **Headline + Subtitle** (last, summarising what was written)

## Example 1 — Loan flow optimisation (B2C banking)

### Context

**⁉️ Problem:** 20% of people who started to ask for a loan were abandoning the flow even before they saw our offer.

**🎯 Target audience:** Clients looking for a loan below 3000€ and looking for transparency

**👯 Team:** 1 technical product manager + 1 iOS dev + 1 Android dev + Me (Product Designer)

**🔑 Key results:**

- +14 points of conversion on main KPI (From 80 to 94%)
- +5,5 points on loan signature rate (From 64,5 to 70%)
- 3 new opportunities detected thanks to improved data tracking

### How I solved this problem?

1. Detection of the opportunity via Amplitude data analysis
2. Mapping of cases and potential paths to make sure to handle all potential cases (loan is based on multiple parameters which can lead to dead-ends if badly managed)
3. Design of an improved UX based
4. Definition of an improved tracking plan
5. Deployment in production
6. Tracking of impact with an hand-made new Amplitude dashboard
7. Iteration on new opportunities

### What did we produce in the end?

- A better converting loan simulation flow
- An improved analytics dashboard which helped us discover 3 new opportunities

## Example 2 — A/B Test metrics (CRO / AB Tasty)

### Context

**⁉️ Problem:** CRO landscape is crowded with tools that provide the exact same benefits but none answer the real question: does this variation makes me earn money in the end?

**🎯 Target audience:** CRO managers

**👯 Team:** 1 Data scientist + 1 Product manager + 1 tech lead + Me (Product designer)

**🔑 Key results:**

- 80% of CRO managers said they will be very disappointed if the feature disappear
- A new unique competitive advantage for AB Tasty salespeople, KAM & CSM

### How I solved this problem?

1. Understand and play with the data generated by our data scientist
2. Challenge data format and push towards a more user-oriented data-approach
3. Build a prototype on Google Sheet to validate user interest and data format at low cost
4. Run user-tests with end clients and true data thanks to our Google Sheet prototype
5. Design proper prototype in Figma
6. Run user-tests with end clients to validate user understanding of these metrics
7. Iterate on prototype and reprioritize investments
8. Participate to internal & external communication about the project
9. Track user satisfaction with user surveys & data analytics

### What did we produce in the end?

- A new set of metrics to take better decisions on A/B test
- An AI-powered insights block that help you read your test results
- An AI-powered chat that allow our clients to chat with their data

## Example 3 — Public API integration (B2B SaaS / PayFit)

### Context

**⁉️ Problem:** While we were building a large set of integrations for our clients, some tools were not widely used enough to become a native integration. We needed to find a scalable solution to this issue.

**🎯 Target audience:** TPE with tech-oriented leaders & PME with custom integration needs

**👯 Team:** 1 group product manager + 1 Tech lead + 3 developers + Me (Product Designer)

**🔑 Key results:** Churn reduction thanks to an increased engagement of clients who did connect with PayFit public API

### How I solved this problem?

1. 5 discovery user interviews
2. Co-internal-discovery with technical team
3. Benchmark among 10 competitors
4. User-flow & design v1
5. User-flow & design v2 to optimize delivery time
6. Documentation architecture design & writing
7. Follow-up with marketing team on communication

### What did we produce in the end?

The answer came in 3 parts:

- A dedicated space in PayFit app to manage API keys and webhooks configurations
- A flow to open a dedicated access to PayFit for people in-charge of technical implementation
- An easy-to-use developer documentation to make clients autonomous (https://developers.payfit.io)

## Implications for the future `/case-study` skill

When generating Section 3 of a mission MD, the skill should:

1. **Skip Headline + Subtitle initially.** Generate the Context block first, then How, then Delivered, then come back to Headline + Subtitle.
2. **Use the compact format** with emoji anchors, not deep H3 nesting.
3. **Surface Key results in Context** (not just at the end).
4. **Keep "How I solved" steps to 1-line action statements** where possible. A single parenthetical clarification is allowed when context is needed.
5. **End with a short "What did we produce" outcome bullet list.**
6. **Generate Headline + Subtitle last**, summarising what was written.
7. **Ask questions one block at a time** (not all upfront), proposing 2-3 candidate phrasings per field for the user to pick / amend.
